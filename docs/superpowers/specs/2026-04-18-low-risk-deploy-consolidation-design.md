# 低风险部署收敛设计

## 目标

在不改变现有运行拓扑、不主动打断线上服务的前提下，把当前可用但分散的部署链路收敛为一套明确、可重复、可回滚的流程。

本次设计只覆盖以下内容：

- 明确唯一部署源码目录
- 把部署步骤脚本化
- 把健康检查标准固定为 HTTP 接口
- 为前端静态产物和后端 JAR 增加可回滚备份
- 把服务器目录、服务、发布和回滚步骤写入仓库文档

本次设计不覆盖以下内容：

- 改造域名、Nginx 拓扑、数据库结构
- 引入 CI 制品发布
- 容器化改造前后端
- 轮换现有账户或密钥

## 当前事实

截至 2026-04-18 调查结果：

- 服务器系统为 Ubuntu 22.04.5 LTS
- Nginx 由宝塔面板管理
- 正式站点由 `https://www.excelcc.cn` 提供
- Nginx 静态根目录为 `/www/wwwroot/kick-web`
- `/api/`、`/uploads/`、`/ws/` 反代到 `127.0.0.1:8080`
- 后端由 `kick-backend.service` 通过 `systemd` 托管
- 后端工作目录为 `/www/wwwroot/kick-backend`
- 服务环境文件为 `/www/wwwroot/kick-backend/.env.production`
- 当前部署源码目录实际为 `/www/wwwroot/kick-deploy/repo`
- `/www/wwwroot/kick-src/kick` 是历史工作目录，不应继续作为发布入口

## 主要问题

### 1. 部署入口不唯一

服务器同时存在：

- `/www/wwwroot/kick-deploy/repo`
- `/www/wwwroot/kick-src/kick`

这会导致“在哪个目录改代码、从哪个目录发布”的边界不清，增加误发布风险。

### 2. 部署过程不显式

日志表明存在部署流程，但脚本本体没有纳入当前仓库。部署依赖操作者记忆，缺乏统一入口和显式参数。

### 3. 成功判定不可靠

现有日志停在“8080 端口未就绪”，但服务随后已恢复运行。这说明仅以端口监听判断成功不够准确。

### 4. 回滚路径不清晰

当前运行目录有备份痕迹，但没有一套固定的“部署前备份、失败时回滚”的明确规则。

### 5. 部署仓处于脏状态

服务器上的 `/www/wwwroot/kick-deploy/repo` 存在未提交修改，不利于未来重复执行部署。

## 方案选择

### 方案 A：低风险收敛

保留现有运行拓扑：

- `Nginx -> /www/wwwroot/kick-web`
- `Nginx -> 127.0.0.1:8080`
- `systemd -> kick-backend.service -> forum-1.0.0.jar`

在此基础上增加：

- 唯一部署目录约定
- 仓库内部署脚本
- 健康检查
- 备份与回滚
- 运维文档

这是本次采用的方案。

### 方案 B：CI 产物化发布

长期更稳，但会引入 GitHub Actions、产物分发和凭据管理，不适合作为这轮最低风险改造。

### 方案 C：容器化重构

改动面过大，与“先不影响正常运行”的目标冲突。

## 目标状态

完成后应满足以下约束：

- 服务器唯一部署源码目录为 `/www/wwwroot/kick-deploy/repo`
- 运维人员从仓库内固定脚本执行部署
- 前端构建产物统一发布到 `/www/wwwroot/kick-web`
- 后端 JAR 统一发布到 `/www/wwwroot/kick-backend/forum-1.0.0.jar`
- 后端重启统一通过 `systemctl restart kick-backend.service`
- 成功判定统一使用 `http://127.0.0.1:8080/api/public/home-overview`
- 每次发布前保留一份可直接回退的前端和后端备份

## 设计细节

### 1. 唯一部署入口

把 `/www/wwwroot/kick-deploy/repo` 定义为唯一部署源码目录。

处理原则：

- 部署脚本只接受该目录作为源码根目录
- 文档只描述该目录
- `/www/wwwroot/kick-src/kick` 仅标记为历史目录，不再参与部署

### 2. 部署脚本职责

仓库内新增生产部署脚本，职责如下：

1. 校验必要目录和命令存在
2. 读取部署配置
3. 检查当前分支和工作树状态
4. 构建前端
5. 构建后端 JAR
6. 创建带时间戳的回滚备份
7. 用新产物替换运行时目录
8. 重启 `kick-backend.service`
9. 用 HTTP 接口做健康检查
10. 失败时输出明确故障点；在可安全回退时执行回滚

### 3. 运行时替换策略

前端：

- 构建目录为 `reace_web/dist`
- 运行目录为 `/www/wwwroot/kick-web`
- 采用“先备份再覆盖”的方式发布

后端：

- 构建产物为 `excel-forum-backend/target/forum-1.0.0.jar`
- 运行产物为 `/www/wwwroot/kick-backend/forum-1.0.0.jar`
- 先备份旧 JAR，再替换为新 JAR

### 4. 健康检查

部署成功标准固定为：

- `curl http://127.0.0.1:8080/api/public/home-overview`
- 返回 `200`

原因：

- 比单纯监听端口更接近真实可用状态
- 能覆盖 Spring Boot 已完成启动、路由已可响应的条件

### 5. 回滚策略

每次部署前在专用备份目录中创建一个时间戳目录，至少包含：

- 上一版 `kick-web`
- 上一版 `forum-1.0.0.jar`
- 对应部署时间

失败处理策略：

- 如果替换后后端健康检查失败，则恢复上一版 JAR 和上一版前端目录
- 回滚后再次重启 `kick-backend.service`
- 回滚结果也要记录

### 6. 文档策略

在仓库中新增一份部署运维文档，明确记录：

- 服务器目录映射
- 服务名称
- 部署命令
- 健康检查命令
- 回滚命令
- 历史目录说明

## 风险与控制

### 风险 1：部署时服务短暂不可用

控制：

- 不调整 Nginx 配置
- 不变更服务名
- 只在产物已构建完成后执行替换和重启
- 用回滚备份缩短恢复时间

### 风险 2：线上脏仓导致脚本失败

控制：

- 部署脚本明确打印工作树状态
- 文档明确要求：线上部署仓不得保留未纳入仓库的临时修改

### 风险 3：构建环境波动

控制：

- 当前阶段接受“服务器本机构建”
- 在文档中记录当前依赖版本
- 为后续切换到 CI 产物化发布留下接口

## 验收标准

完成后应能验证：

1. 仓库内存在明确部署脚本
2. 仓库内存在明确部署文档
3. 服务器发布入口收敛到 `/www/wwwroot/kick-deploy/repo`
4. 运行服务仍由 `kick-backend.service` 托管
5. 执行部署后，`https://www.excelcc.cn` 与 `/api/public/home-overview` 正常
6. 发生失败时可按文档执行回滚

## 后续阶段

本轮完成后，下一阶段优先项为：

1. 把生产构建迁移到 CI
2. 关闭数据库公网暴露或收敛访问源
3. 增加后端健康端点与更细粒度监控
